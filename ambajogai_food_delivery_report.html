Referral economics (precise)

Goal: give coins only for orders ₹150+, maintain an overall referral budget capped at 5% of the item order value, avoid cash payouts to reduce abuse and legal risk.

Distribution model

  • Total referral budget per qualifying order = 5% × item_price
  • Distribution across up to 5 levels (configurable). For a safe, conservative start we recommend 2 levels live and reserve levels 3–5 for future if fraud is controlled. Example split (when using 5 levels) — L1=50%, L2=20%, L3=15%, L4=10%, L5=5% (applied to the referral budget).

Worked example (₹250 order — qualifies)

Item price = ₹250
Referral budget = 5% × 250 = ₹12.50
Distribution example (5-level):
  L1 (50%) = 0.5 × 12.5 = ₹6.25 -> round to ₹6
  L2 (20%) = 0.2 × 12.5 = ₹2.50 -> ₹3
  L3 (15%) = 1.875 -> ₹2
  L4 (10%) = 1.25 -> ₹1
  L5 (5%) = 0.625 -> ₹1 (or ₹0 depending on rounding rule)
Total ≈ ₹13 (rounding small differences handled by platform reserve)
Notes: coins issued are 'platform-credit' with monetary equivalence for UX only; coins are redeemable as discounts on future orders only. Coins expire in 90 days by default.
Important: because coins are non-withdrawable and usable only against purchases, they count as discounts (marketing expense) rather than cash payouts. This reduces legal/regulatory scrutiny compared to cash commissions across levels.

Platform money flow (one order)

  1. Customer pays: item_price (P) + delivery_fee (D) + taxes.
  2. Payment gateway collects funds and settles net to platform (or using split-payments, splits to merchant & platform according to rules).
  3. From item_price P:
    • Merchant receives 70% of P (after marketplace settlement & taxes).
    • Referral pool receives 5% of P (credited as coins to qualifying referrers according to referral ledger rules).
    • Platform retains 25% of P (20% commission + 5% maintenance) as revenue.
  4. Delivery fee D: collected from customer; platform keeps handling fee (₹5–10) and pays remaining D to rider (or to courier partner if outsourced).

Technical architecture (MERN, minimal components)

Simple, scalable components for a local-first web-only MVP (PWA-ready):

  • Frontend: React + Tailwind CSS (PWA support for offline & web-push). Single codebase deployed to static hosting (CDN) with an API server.
  • Backend: Node.js + Express (REST) + Socket.IO for real-time order updates.
  • Database: MongoDB Atlas (start with free/low-tier cluster). Use collections: users, merchants, orders, riders, referralLedger, settlements.
  • Cache / Jobs: Redis for caching, rate-limits & short locks. Use a task queue (BullMQ or similar) for background jobs (settlement, email/SMS batching).
  • Storage: S3-compatible (DigitalOcean Spaces / AWS S3) for images and receipt files.
  • Third parties: Payment Gateway (Razorpay/Cashfree), Maps (Mapbox or Google Maps), SMS (MSG91), Push (Firebase Cloud Messaging), Email (SendGrid).
  • Deployment: Docker + Nginx reverse-proxy. Start with 1-2 droplets (or small VM instances) and upgrade as usage grows. Optionally use managed Kubernetes if you expect rapid scaling.

  1. Register entity (LLP or Pvt Ltd). Budget ₹10k–₹40k for registration + professional fees.
  2. GST registration if threshold met or operating interstate; implement GST invoicing in settlements.
  3. Merchant FSSAI verification: require merchants to upload FSSAI during onboarding.
  4. Payment gateway KYC & bank details for merchant payouts (Razorpay / Cashfree onboarding).
  5. Referral T&Cs: publish clear rules — coins non-withdrawable, expire in 90 days, coins only for qualifying orders (₹150+), and anti-abuse clause.

Go-to-market & merchant acquisition (practical)

  1. Identify 8–12 anchor merchants (popular local restaurants) and offer 0% commission for first 30 days or waived onboarding fee to create supply-side liquidity.
  2. Offer riders guaranteed first-week incentives to ensure fast deliveries at launch.
  3. Local promotions: college discounts, festival offers, posters, reseller merchant cross-promos.
  4. Track activation & retention metrics: % of customers who use coins within 30 days, 1st-to-2nd order retention, merchant repeat orders/week.

Fraud controls & safety

  • OTP + device fingerprinting + rate limits at signup to prevent multi-account creation.
  • Referral anti-abuse: reject same-device or same-IP chain referrals for coin issuance; flag suspicious accounts for manual review.
  • Limit coin issuance per referrer per month (e.g., max ₹2,000 in coins) to curb abuse.
  • Require first successful non-refunded delivery to unlock referral earnings for referrer.
